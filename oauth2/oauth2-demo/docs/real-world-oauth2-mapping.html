2 Step-by-Step Flow

The OAuth2 protocol is identical in both cases. Every Google step has a demo counterpart.

0

Prerequisites — Registering the Client

🔵 Google Cloud Console

Developer goes to APIs & Services → Credentials → Create OAuth 2.0 Client ID

Fields: Client ID, Client Secret, Authorized Redirect URIs, Scopes

🟢 Demo

The RegisteredClient is seeded programmatically at startup in AuthorizationServerConfig.registeredClientRepository().

Java.clientId("demo-client")
.clientSecret(passwordEncoder.encode("secret"))
.redirectUri("http://localhost:3000/callback")
.scope("openid", "profile")
1

User Clicks "Sign In"

🔵 Google

User clicks "Sign in with Google" on a third-party website. The website constructs the authorization URL with client_id, redirect_uri, scope, and PKCE params.

🟢 Demo

User clicks "Login with OAuth2" in the React frontend. App.js → handleLogin() generates PKCE challenge and constructs the URL.

Google URL# Real-World (Google) Authorization URL
https://accounts.google.com/o/oauth2/v2/auth?
  response_type=code&
  client_id=738...apps.googleusercontent.com&
  redirect_uri=https://myapp.com/callback&
  scope=openid%20profile%20email&
  code_challenge=E9...&
  code_challenge_method=S256&
  state=abc123
⬇
Demo URL# Demo Analogous URL
http://localhost:8085/oauth2/authorize?
  response_type=code&
  client_id=demo-client&
  redirect_uri=http://localhost:3000/callback&
  scope=openid%20profile&
  code_challenge=E9...&
  code_challenge_method=S256&
  state=random-state-xyz
2

User Authenticates

🔵 Google Login

Google displays its login page — user enters Gmail email + password. Google validates against Google Accounts (millions of users, MFA).

🟢 Demo Login

Spring AS displays the built-in form login page at /login. Validates against InMemoryUserDetailsManager with 2 hardcoded users:

  • user / password — ROLE_USER
  • admin / admin — ROLE_ADMIN
💡 Key difference: Google uses a massive distributed identity platform. The demo uses a simple in-memory store — but the OAuth2 protocol is identical. You could swap in a JdbcUserDetailsManager or LDAP without changing any OAuth2 logic.
3

User Consents to Scopes

🔵 Google Consent

"myapp.com wants to: View your email address, View your profile info"
User clicks "Allow". Scopes: openid, profile, email

🟢 Demo Consent

Consent page (because requireAuthorizationConsent=true):
"Demo Client requests: ☐ openid ☐ profile"
User clicks "Submit Consent". Scopes: openid, profile

💡 Google often skips the consent screen for trusted apps. Set requireAuthorizationConsent=false to mirror Google's behavior.
4

Authorization Code Issued

🔵 Google Redirect

Google redirects the browser back to the third-party site with a one-time-use authorization code bound to the PKCE challenge. state prevents CSRF.

🟢 Demo Redirect

Same — the code is stored in the oauth2_authorization table (H2) bound to the code_challenge. state=random-state-...

HTTP 302# Redirect Response (both Google and Demo)
302 Redirect
Location: https://myapp.com/callback?code=4/0AX...&state=abc123
5

Token Exchange (Back-Channel)

🔵 Google (server-to-server)

The third-party backend makes a server-to-server POST to Google. Client secret is kept safe on the server.

🟢 Demo (client-side)

The React frontend makes a direct POST from the browser (CORS-enabled).

Google# Google Token Request (server-side)
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=4/0AX...&
redirect_uri=https://myapp.com/callback&
client_id=738...apps.googleusercontent.com&
client_secret=GOCSPX-...&
code_verifier=dBj...
⬇
Demo# Demo Token Request (client-side)
POST http://localhost:8085/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=abc123...&
redirect_uri=http://localhost:3000/callback&
client_id=demo-client&
client_secret=secret&
code_verifier=dBj...
💡 Key difference: Google's token exchange is server-side. In this demo it's client-side for simplicity. A production setup would proxy through the backend.

7

Optional — Token Refresh

🔵 Google Refresh

When access token expires (3600s), the backend uses the refresh_token to get a new one.

POST https://oauth2.googleapis.com/token
grant_type=refresh_token&
refresh_token=1//0g...&
client_id=738...apps.googleusercontent.com

🟢 Demo Refresh

The demo supports REFRESH_TOKEN grant type — included in the provider chain.

OAuth2AuthorizedClientProviderBuilder.builder()
    .authorizationCode()
    .refreshToken()   // ← enabled
    .clientCredentials()
    .build();

4 Security Comparison

How security measures compare between Google's production system and the local demo.

🔏 Token Signing

Google: Publicly known RSA keys rotated periodically

Demo: RSA-2048 key generated at each startup — existing tokens invalidated on restart

📌 PKCE

Google: Required for all public clients (since 2021)

Demo: S256 enforced — frontend implements full PKCE

🔒 Client Secret

Google: Stored in server config, never exposed to browser

Demo: Embedded in frontend for simplicity (would be backend-only in production)

🔐 TLS/HTTPS

Google: Required — all endpoints enforce HTTPS

Demo: HTTP — localhost only. In production TLS is mandatory.

👤 User Store

Google: Google Accounts — billions of users, MFA, security monitoring

Demo: InMemoryUserDetailsManager — 2 users, no MFA

✅ Consent

Google: Shown once, remembered per client

Demo: Shown every time (requireAuthorizationConsent=true) — can be changed to false

📄 Token Format

Google: Opaque access tokens + JWT ID tokens

Demo: JWT access tokens + JWT ID tokens (both RS256)

🔄 Refresh Tokens

Google: Long-lived, can be revoked by user

Demo: Supported but expires when H2 resets on restart

🛡️ Client Registration

Google: Google Cloud Console — managed UI

Demo: Hardcoded @Bean — seeded at startup in H2

5 What Would Need to Change to Be "Production-Like"

Making the demo behave like a real Google-style identity provider.

Key managementKey generated at each startupPersistent key store (e.g., JWK Set loaded from file / Vault)
HTTPSPlain HTTPTLS certificate

💡 The Best Part

The OAuth2 protocol code itself doesn't change when moving from demo to production. You swap out the infrastructure (database, TLS, key management) while keeping the same Spring Security configuration. The same code that integrates with Google's AS can integrate with a local Spring AS by changing only the ClientRegistration properties (issuer URI, client ID, client secret).

Why OAuth2 Is Powerful

  • Scale: Google serves billions; this demo serves one developer — same protocol
  • URLs: accounts.google.com → localhost:8085 — same structure
  • Users: Google Accounts → 2 hardcoded test users — same API
  • Keys: Google's managed PKI → a one-line RSA key generator — same algorithms
  • Portability: Switch from Google AS to a local Spring AS by changing only issuer-uri, client-id, and client-secret
